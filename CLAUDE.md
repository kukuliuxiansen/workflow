# CLAUDE.md - Workflow Engine Project

## 核心原则
- **KISS 原则**: 保持简单，拒绝过度设计。
- **DRY 原则**: 严禁复制粘贴代码，重复逻辑必须提取为公共函数。
- **文件限制**: 单个文件最大 200 行。超出必须拆分。

## 代码规模与重构原则

1. **软性限制**：单个源文件（不含测试文件）建议保持在 **200-250 行** 以内。
2. **触发重构**：当添加新功能导致文件超过 **300 行** 时，**禁止**直接追加代码。必须执行以下操作之一：
   - **提取函数/组件**：将独立的逻辑块提取为私有函数或子组件。
   - **模块拆分**：如果文件包含多个不相关的职责（例如同时处理“数据获取”和“UI 渲染”），必须按职责拆分为多个文件。
   - **工具类提取**：将通用的辅助逻辑移动到 `utils/` 或 `helpers/` 目录。
3. **避免过度拆分**：
   - 不要为了拆分而拆分。如果两个部分逻辑强耦合（例如父组件强依赖子组件的内部状态），允许暂时保留在同一个文件中，但需添加 `// TODO: 考虑拆分` 注释。
   - 单个文件不应小于 20 行（除非是纯类型定义或配置），避免产生大量碎片文件。


## 修改规则
- 在修改现有代码前，必须先分析该函数的调用链，确保不影响下游。
- 禁止直接修改核心底层库文件，如有需求需通过适配器模式扩展。



## 🛡️ 核心协作宪章 (Core Collaboration Charter)

### 1. 绝对诚实与零容忍谎言 (Absolute Honesty Policy)
- **禁止虚构**：严禁编造不存在的库、函数、API 端点、配置文件或系统环境。如果不确定某个功能是否支持，必须明确回答“我不确定”或“需要验证”，绝不可猜测。
- **禁止虚假承诺**：
  - 如果任务超出当前能力或上下文限制，必须立即声明，不得假装完成。
  - 如果代码无法在当前环境下运行，必须说明缺失的依赖或前置条件。
- **事实核查**：在引用外部文档或现有代码逻辑时，必须基于实际读取的文件内容，不得依赖训练数据中的过时信息。

### 2. 强制交付原则 (Mandatory Completion)
- **有始有终**：一旦接受任务，必须完整实现所有需求，直到功能可运行为止。
  - 禁止使用 `// ... rest of code` 或 `<!-- 省略部分 -->` 等占位符。必须输出完整的、可运行的代码文件。
  - 如果因长度限制被截断，必须主动提示用户“继续”，并补全剩余代码。
- **闭环验证**：任务完成的定义不仅仅是“代码写完”，而是“代码已验证可用”。

### 3. 实证主义：测试与日志 (Empiricism: Test & Logs)
**这是本项目的最高优先级规则。没有证据的代码视为未完成。**

#### A. 必须执行测试 (Must Run Tests)
- **自动测试**：实现任何新功能或修复 Bug 后，**必须**编写对应的单元测试或集成测试，并在回复中展示**测试运行的真实输出结果**（Pass/Fail 日志）。
- **禁止空谈**：不能说“测试通过了”，必须贴上类似以下的真实终端输出


## 🎲 动态验证机制
- **动态指纹**：在所有测试脚本和日志输出中，必须包含至少一个**动态生成的指纹**（如：当前精确到毫秒的时间戳 `date +%s%N`、随机 UUID、或进程 PID）。
- **禁止静态模板**：日志中严禁出现固定的、重复的“假时间”（如永远是 `12:00:00`）或固定的“假ID”。
- **验证指令**：如果用户要求验证，必须能够重新运行命令并生成**不同**的动态指纹，以证明代码是实时执行的。
