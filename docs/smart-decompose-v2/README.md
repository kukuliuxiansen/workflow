## 一、概述

### 1.1 核心理念

**SmartDecompose = 项目经理**

- 不自己执行具体操作
- 负责任务分解、编排、验收
- 委托 OpenClaw 执行具体工作

### 1.2 核心能力

```
大任务 → 决策（执行/拆分）→ 执行 → 审核 → 完成
```

| 能力 | 说明 |
|-----|------|
| 智能决策 | 判断任务是否需要拆分 |
| 任务分解 | 复杂任务拆成多个子任务 |
| 执行委托 | 调用 OpenClaw 执行 |
| 结果审核 | 调用 OpenClaw 审核 |
| 失败重试 | 审核不通过自动重试 |
| 人工介入 | 重试失败后人工确认 |

### 1.3 与 OpenClaw 的关系

```
┌─────────────────┐         ┌─────────────────┐
│ SmartDecompose  │         │    OpenClaw     │
│   (项目经理)     │         │    (执行者)      │
├─────────────────┤         ├─────────────────┤
│ • 任务分解      │ ──────> │ • 读取文件      │
│ • 流程编排      │         │ • 写入文件      │
│ • 结果验收      │ <────── │ • 执行命令      │
│ • 失败重试      │         │ • 搜索代码      │
│ • 人工介入      │         │ • ...           │
└─────────────────┘         └─────────────────┘
```

## 三、关键设计决策

### 3.1 提示词存储

| 内容 | 存储位置 | 原因 |
|-----|---------|------|
| 提示词模板 | 数据库 | 需要灵活调整，不同场景不同模板 |
| JSON 输出格式 | 代码中 | 固定不变，硬编码更可靠 |
| 上下游节点信息 | 代码中 | 动态生成，不适合存储 |

### 3.2 执行模式

| 模式 | 说明 |
|-----|------|
| 完全串行 | OpenClaw 只支持单会话，子任务必须串行执行 |
| 递归拆分 | 子任务仍可继续拆分，直到可执行粒度 |

### 3.3 审核机制

```
执行 → 审核 → 通过 → 下一个任务
              ↓
           拒绝 → 重试(最多5次) → 人工介入
```

## 四、快速理解

### 4.1 一个例子

**输入**: "开发一个登录网站"

**第1轮 - 决策**:
```json
{
  "decision": "split",
  "tasks": [
    {"id": "TASK_001", "description": "创建项目结构"},
    {"id": "TASK_002", "description": "实现登录Controller"},
    {"id": "TASK_003", "description": "创建登录页面"}
  ]
}
```

**第2轮 - 执行 TASK_001**:
```json
{
  "decision": "execute",
  "result": "已创建 pom.xml 和主类文件"
}
```

**第2轮 - 审核**:
```json
{
  "status": "APPROVED",
  "summary": "项目结构创建成功"
}
```

**第3-4轮**: 执行 TASK_002、TASK_003...

**最终输出**: 所有任务完成，返回成功结果
