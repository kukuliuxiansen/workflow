## P0 - 紧急问题

### 1. 任务配置与工作流绑定混乱

**问题文件**: `task-config.js:53-62`

**问题描述**:
- 任务配置保存到工作流，但执行时又从 `taskConfig` 全局变量读取
- 切换工作流后，任务配置不会自动切换
- 可能导致用错误的配置执行工作流

---

#### 功能需求描述

任务配置（任务名称、描述、项目路径）是执行工作流的必要参数。**每个工作流应该有独立的任务配置**，切换工作流时应自动切换对应的配置。

---

#### 问题举例

**场景**: 用户管理两个不同项目的工作流

| 步骤 | 操作 | 期望行为 | 实际行为 |
|------|------|----------|----------|
| 1 | 选择工作流A，配置任务路径为 `/project/a` | 配置保存到工作流A | ✅ 正常 |
| 2 | 切换到工作流B，配置任务路径为 `/project/b` | 配置保存到工作流B | ✅ 正常 |
| 3 | 切换回工作流A | 自动加载A的配置 `/project/a` | ❌ 仍显示 `/project/b` |
| 4 | 直接点击执行 | 使用A的配置执行 | ❌ 使用B的配置执行 |

**后果**: 代码生成到错误的项目目录，可能覆盖其他项目的文件。

---

#### 正确的交互流程

```
用户选择工作流A
    ↓
检查工作流A是否有已保存的任务配置
    ↓
├── 有配置 → 加载到 taskConfig 全局变量 → 更新UI显示
└── 无配置 → 清空 taskConfig → 显示"请先配置任务"
    ↓
用户修改配置并保存
    ↓
配置保存到工作流A的 taskConfig 字段
    ↓
用户切换到工作流B
    ↓
重复上述流程（加载B的配置或清空）
```

---

#### 实现方案

**修改文件**: `workflow-loader.js`

**修改方法**: `selectWorkflow()`

- 加载工作流后，读取工作流的 `taskConfig` 字段
- 若存在，赋值给全局 `taskConfig` 变量
- 若不存在，重置 `taskConfig` 为空对象
- 调用 `updateTaskConfigDisplay()` 更新工具栏显示

**修改文件**: `task-config.js`

**修改方法**: `saveTaskConfig()`

- 保存成功后，同步更新 `state.currentWorkflow.taskConfig`
- 保持现有的保存到工作流逻辑不变

**修改文件**: `global-config.js`

**新增方法**: `resetTaskConfig()`

- 清空 `taskConfig` 全局变量的所有字段
- 用于切换到未配置的工作流时调用

### 2. 执行前验证不完整

**问题文件**: `workflow-save.js:36-65`

**问题描述**:
- 执行前仅检查开始/结束节点和任务配置
- 未检查节点是否正确配置、连线是否完整
- 导致执行中途失败，浪费时间和资源

---

#### 功能需求描述

工作流执行前应进行**完整的前置检查**，确保所有节点都已正确配置，连线完整无孤立节点，条件分支和循环逻辑正确。验证失败应给出明确的错误提示和定位。

---

#### 问题举例

**场景1**: Agent节点未选择Agent

```
工作流: 数据处理流程
节点: [开始] → [Agent执行: 数据清洗] → [结束]

问题: "数据清洗"节点未选择 Agent ID
结果: 执行到该节点时报错 "Agent ID 不能为空"
```

**期望**: 点击执行前，提示 "节点【数据清洗】未配置 Agent"

---

**场景2**: 节点孤立未连线

```
工作流:
  [开始] → [Agent执行: 步骤A] → [结束]
  [Agent执行: 步骤B]  (孤立节点，未连线)

问题: 步骤B节点存在但未连接到流程中
结果: 步骤B永远不会执行，但用户不知道
```

**期望**: 验证时提示 "节点【步骤B】未连接到流程"

---

**场景3**: 条件节点分支不完整

```
工作流:
  [开始] → [条件判断: 检查结果] → [结束]
            ↓ (仅有成功分支)
          [Agent执行: 处理成功]
            ↓
          [结束]

问题: 条件节点没有失败分支
结果: 条件失败时流程卡住或报错
```

**期望**: 验证时提示 "条件节点【检查结果】缺少失败分支"

---

#### 正确的验证流程

```
用户点击"执行"按钮
    ↓
第一步：结构验证
├── 开始节点唯一性检查
├── 结束节点存在性检查
└── 所有节点都已连线（无孤立节点）
    ↓
第二步：节点配置验证
├── Agent节点：必须选择 Agent ID
├── API节点：必须填写 URL
├── 条件节点：至少有一个出边
└── 循环节点：必须配置最大循环次数
    ↓
第三步：流程逻辑验证
├── 从开始节点能到达所有节点
├── 所有节点都能到达结束节点
└── 不存在无限循环（循环必须有退出路径）
    ↓
第四步：任务配置验证
├── 任务名称已填写
├── 任务描述已填写
└── 项目路径已填写且有效
    ↓
全部通过 → 弹出确认框 → 执行
任一失败 → 显示错误列表 → 定位问题节点
```

---

#### 实现方案

**修改文件**: `workflow-save.js`

**新增方法**: `validateWorkflowComplete()`

- 返回对象：`{ valid: boolean, errors: [], warnings: [] }`
- 错误对象格式：`{ nodeId: string, message: string, type: string }`
- 依次执行结构验证、节点配置验证、流程逻辑验证

**修改方法**: `executeWorkflow()`

- 在现有验证前调用 `validateWorkflowComplete()`
- 若验证失败，显示错误列表弹窗，不执行
- 点击错误项可定位到对应节点

**修改文件**: `property-panel.js`

**修改方法**: `renderAgentFields()`

- Agent ID 未选择时，节点显示警告标识
- 可选：在节点卡片上显示"未配置"状态

**新增UI组件**: 验证结果弹窗

- 列出所有错误和警告
- 错误项可点击跳转到对应节点
- 提供"忽略警告继续执行"选项（仅对 warnings）

### 3. 执行控制与历史管理功能缺失

**问题文件**: `workflow-save.js:103-113`, `execution-history.js`

**问题描述**:
- 工具栏执行按钮仅有"执行"和"停止"，缺少"暂停"和"继续"
- 执行历史面板缺少删除功能
- 执行历史详情缺少运行时操作按钮（暂停/继续/停止）
- 后端API已支持暂停/恢复，但前端未对接

---

#### 功能需求描述

工作流执行是长时间运行的操作，用户需要**完整的执行生命周期控制**：

| 操作 | 说明 | 适用状态 |
|------|------|----------|
| 执行 | 启动新的执行 | 空闲状态 |
| 暂停 | 暂停当前执行，保留现场 | 执行中 |
| 继续 | 从暂停点恢复执行 | 已暂停 |
| 停止 | 终止执行，不可恢复 | 执行中/已暂停 |

同时，执行历史记录需要支持**删除**功能，清理不再需要的记录。

---

#### 问题举例

**场景1**: 发现配置错误需要暂停修改

```
用户点击"执行"
    ↓
工作流开始运行，执行到第3个节点
    ↓
用户发现项目路径配置错误
    ↓
用户想要暂停 → 修改配置 → 继续执行
    ↓
❌ 没有"暂停"按钮，只能"停止"后重新执行
❌ 已执行的节点需要重新跑一遍
```

**期望**: 点击"暂停" → 修改配置 → 点击"继续" → 从断点恢复

---

**场景2**: 执行历史无法清理

```
用户测试工作流，产生了10条执行记录
    ↓
用户想要删除测试记录，只保留正式执行的记录
    ↓
❌ 执行历史面板没有删除按钮
❌ 记录越来越多，无法清理
```

**期望**: 每条记录右侧有删除按钮，支持批量删除

---

**场景3**: 从历史记录恢复执行状态

```
用户打开执行历史面板
    ↓
选择一条"已暂停"状态的记录
    ↓
用户想要继续执行
    ↓
❌ 历史详情只有"重新执行"按钮
❌ 无法"继续"这条暂停的记录
```

**期望**: 根据记录状态显示对应的操作按钮（继续/停止）

---

#### 正确的交互流程

**工具栏按钮状态流转**:

```
[空闲状态]
├── 显示：[执行] 按钮
└── 隐藏：[暂停] [继续] [停止] 按钮

    ↓ 点击执行

[执行中状态]
├── 显示：[暂停] [停止] 按钮
└── 隐藏：[执行] [继续] 按钮

    ↓ 点击暂停

[已暂停状态]
├── 显示：[继续] [停止] 按钮
└── 隐藏：[执行] [暂停] 按钮

    ↓ 点击继续

[执行中状态]（回到上面）

    ↓ 点击停止 或 执行完成

[空闲状态]（回到最上面）
```

**执行历史面板操作按钮**:

| 记录状态 | 可用操作 |
|----------|----------|
| 执行中 | 暂停、停止 |
| 已暂停 | 继续、停止 |
| 已完成 | 重新执行、删除 |
| 已停止 | 重新执行、删除 |
| 已失败 | 重新执行、删除 |

---

#### 实现方案

**修改文件**: `index.html`

**修改位置**: 工具栏区域（`.toolbar`）

- 新增 `[暂停]` 按钮，默认隐藏
- 新增 `[继续]` 按钮，默认隐藏
- 调整按钮显示逻辑，根据执行状态切换

**修改文件**: `workflow-save.js`

**修改方法**: `executeWorkflow()`

- 执行成功后，隐藏执行按钮，显示暂停和停止按钮
- 更新 `state.executionStatus` 状态

**新增方法**: `pauseExecution()`

- 调用后端 API：`POST /api/executions/{id}/pause`
- 成功后切换按钮状态：隐藏暂停，显示继续

**新增方法**: `resumeExecution()`

- 调用后端 API：`POST /api/executions/{id}/resume`
- 成功后切换按钮状态：隐藏继续，显示暂停

**修改方法**: `stopExecution()`

- 停止成功后，恢复到空闲状态
- 显示执行按钮，隐藏其他按钮

**修改方法**: `resetExecution()`

- 重置所有按钮状态
- 更新 `state.executionStatus`

**修改文件**: `execution.js`

**修改方法**: WebSocket 消息处理 `ws.onmessage`

- 收到 `status` 消息时，同步更新按钮状态
- 支持服务端推送的状态变更（如：人工审核暂停）

**修改文件**: `execution-history.js`

**修改方法**: `renderHistoryDetail()`

- 根据记录状态动态显示操作按钮
- 已暂停：显示"继续执行"、"停止"
- 执行中：显示"暂停"、"停止"
- 已完成/已停止/已失败：显示"重新执行"、"删除"

**新增方法**: `deleteExecutionRecord(executionId)`

- 弹出确认框
- 调用后端 API 删除记录
- 刷新历史列表

**修改文件**: `node-exec-actions.js`

**修改方法**: `restartExecution()`

- 重启成功后，同步更新工具栏按钮状态

**新增后端API**: `DELETE /api/executions/records/{executionId}`

- 删除执行记录及相关数据
- 返回删除结果

**修改文件**: `state.js`

**新增状态**: `executionStatus`

- 可能值：`idle`（空闲）、`running`（执行中）、`paused`（已暂停）
- 用于控制按钮显示状态

### 4. 执行进度与节点状态不可见

**问题文件**: `execution-history.js:48-103`, `canvas-render.js`

**问题描述**:
- 执行中只能看日志滚动，不知道执行到哪个节点
- 切换执行历史记录时，节点状态不会渲染到画布上
- 用户无法直观看到工作流的整体执行情况

---

#### 功能需求描述

**节点状态可视化**是工作流执行的核心能力：

| 节点状态 | 颜色 | 说明 |
|----------|------|------|
| 执行中 | 🔵 蓝色 | 当前正在执行的节点 |
| 成功 | 🟢 绿色 | 执行成功的节点 |
| 失败 | 🔴 红色 | 执行失败的节点 |
| 未执行 | 默认色 | 尚未执行的节点 |

**核心场景**：
1. 执行过程中，实时看到哪个节点在执行
2. 切换历史记录时，看到该次执行的节点状态分布
3. 执行失败时，一眼定位到失败的节点

---

#### 问题举例

**场景1**: 执行中不知道进度

```
用户点击"执行"
    ↓
工作流有10个节点
    ↓
用户只看到日志滚动，不知道执行到第几个了
    ↓
❌ 心里没底，不知道还要等多久
❌ 不知道当前在执行哪个节点
```

**期望**: 画布上当前执行的节点显示蓝色，已完成的显示绿色

---

**场景2**: 切换历史记录后状态丢失

```
用户打开执行历史面板
    ↓
选择一条"已失败"的历史记录
    ↓
❌ 画布上所有节点都是默认颜色
❌ 用户不知道哪个节点失败了
❌ 需要去翻日志找错误原因
```

**期望**: 切换历史记录后，失败节点显示红色，成功节点显示绿色

---

**场景3**: 无法快速定位失败节点

```
工作流有15个节点，执行失败
    ↓
日志显示"节点执行失败"
    ↓
❌ 用户需要翻日志找是哪个节点
❌ 没有直观的视觉定位
```

**期望**: 失败节点红色高亮，一眼就能看到问题在哪

---

#### 正确的交互流程

**执行过程中的实时渲染**:

```
用户点击"执行"
    ↓
WebSocket 连接建立
    ↓
收到 nodeStatus 消息 { nodeId, status }
    ↓
更新 state.nodeStatus Map
    ↓
调用 renderCanvas() 重绘画布
    ↓
节点根据状态显示对应颜色
    ├── running → 蓝色边框 + 动画
    ├── success → 绿色边框 + ✓ 标记
    └── failed → 红色边框 + ✕ 标记
```

**切换历史记录时的状态渲染**:

```
用户在执行历史面板选择一条记录
    ↓
调用 selectHistoryItem(executionId)
    ↓
请求后端获取该执行记录的节点状态
    GET /api/executions/{executionId}/node-statuses
    ↓
返回数据: { nodeId: status, ... }
    ↓
更新 state.nodeStatus Map
    ↓
调用 renderCanvas() 重绘画布
    ↓
节点显示该次执行的最终状态
```

---

#### 节点状态视觉效果

```
┌─────────────────────────┐
│  🤖 数据处理            │  ← 默认（灰色边框）
│  Agent: agent-001       │
└─────────────────────────┘

┌─────────────────────────┐
│  🤖 数据处理    ⚡ 执行中│  ← 蓝色边框 + 动画闪烁
│  Agent: agent-001       │
└─────────────────────────┘

┌─────────────────────────┐
│  🤖 数据处理      ✓ 成功 │  ← 绿色边框 + 成功标记
│  Agent: agent-001       │
└─────────────────────────┘

┌─────────────────────────┐
│  🤖 数据处理      ✕ 失败 │  ← 红色边框 + 失败标记
│  Agent: agent-001       │
└─────────────────────────┘
```

---

#### 实现方案

**修改文件**: `state.js`

**新增状态**: `nodeStatuses`

- 类型：`Map<string, string>`
- 存储：`{ nodeId -> status }` 映射
- 状态值：`pending`、`running`、`success`、`failed`

**修改文件**: `canvas-render.js`

**修改方法**: `renderCanvas()`

- 渲染节点时检查 `state.nodeStatuses.get(node.id)`
- 根据状态添加对应 CSS 类：`node-running`、`node-success`、`node-failed`
- 状态节点添加状态徽章显示

**新增CSS类**: `css/modules/node.css`

- `.node-running`：蓝色边框 + 脉冲动画
- `.node-success`：绿色边框 + 成功图标
- `.node-failed`：红色边框 + 失败图标

**修改文件**: `execution.js`

**修改方法**: WebSocket `onmessage`

- 收到 `nodeStatus` 消息时，更新 `state.nodeStatuses`
- 调用 `renderCanvas()` 重绘
- 可选：添加日志 "节点 [xxx] 执行成功/失败"

**修改文件**: `execution-history.js`

**修改方法**: `selectHistoryItem()`

- 加载历史详情后，请求节点状态数据
- 更新 `state.nodeStatuses`
- 调用 `renderCanvas()` 重绘画布

**新增后端API**: `GET /api/executions/{executionId}/node-statuses`

- 返回该执行记录中所有节点的最终状态
- 格式：`{ nodeId: "success", nodeId2: "failed", ... }`

**修改文件**: `workflow-loader.js`

**修改方法**: `selectWorkflow()`

- 切换工作流时，清空 `state.nodeStatuses`
- 重置画布节点为默认状态
